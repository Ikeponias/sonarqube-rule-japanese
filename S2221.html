Exception を捕捉することは、起こりうる複数の例外を処理する効率的な方法のように思われます。しかし、残念ながら、チェックされた例外と実行時例外の両方を含むすべての例外タイプを捕捉するため、あまりにも広い網をかけることになります。実際、実行時例外も捕捉することが開発者の意図だったのだろうか？誤解を避けるために、チェックされた例外と実行時例外の両方を捕捉することが本当に期待されているならば、catch 節で明示的にリストアップする必要があります。

このルールは、Exception がtry ブロックのメソッドによって明示的にスローされていないときにキャッチされた場合に問題を発生させるものです。

コンプライアンス違反のコード例

try {
  // do something that might throw an UnsupportedDataTypeException or UnsupportedEncodingException
} catch (Exception e) { // Noncompliant
  // log exception ...
}

コンプライアンス・ソリューション

try {
  // do something
} catch (UnsupportedEncodingException|UnsupportedDataTypeException|RuntimeException e) {
  // log exception ...
}

または、実行時例外を捕捉すべきでない場合：

try {
  // do something
} catch (UnsupportedEncodingException|UnsupportedDataTypeException e) {
  // log exception ...
}
